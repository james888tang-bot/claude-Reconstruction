# 致命阻塞规则

> **原则**: 只在这4种情况下提问，其他情况自行决策

---

## 🚨 唯一允许提问的4种情况

### 1️⃣ 缺少关键凭证

**定义**: 必须的敏感信息，无法猜测或推断

**示例**:

- ❌ 数据库密码
- ❌ API key
- ❌ 私钥文件路径
- ❌ 服务器 IP 地址

**如何提问**:

```
需要 [具体信息] 才能继续：
- [信息名称]
- 用途：[为什么需要]
```

---

### 2️⃣ 多个对立方案

**定义**: 2+ 个方案完全对立，无法从代码库或上下文判断

**示例**:

- ❌ 使用 REST 还是 GraphQL？（架构级决策）
- ❌ 数据存 MySQL 还是 PostgreSQL？（技术选型）
- ❌ 用户认证用 OAuth 还是 JWT？（安全策略）

**不是对立的**:

- ✅ 函数命名风格（看代码库现有风格）
- ✅ 文件组织方式（看项目结构）
- ✅ UI 颜色选择（看设计规范或自行选择）

**如何提问**:

```
有2个对立方案，需要决策：

方案A: [描述]
  优点: [...]
  缺点: [...]

方案B: [描述]
  优点: [...]
  缺点: [...]

推荐: [方案X]，因为 [理由]

是否继续？
```

---

### 3️⃣ 需求本质矛盾

**定义**: 用户的多个要求之间逻辑冲突

**示例**:

- ❌ "要快速开发，但每行代码都要完美"
- ❌ "要最大安全性，但不要验证用户输入"
- ❌ "要实时同步，但不能用 WebSocket"

**如何提问**:

```
发现需求矛盾：

要求A: [描述]
要求B: [描述]

这两者冲突，因为 [原因]

可能的折中方案:
- [方案1]
- [方案2]

请明确优先级。
```

---

### 4️⃣ 不可逆高风险

**定义**: 一旦执行无法撤销，且后果严重

**示例**:

- ❌ 删除生产数据库表
- ❌ 强制推送到 main 分支（覆盖历史）
- ❌ 删除整个代码目录
- ❌ 撤销已发布的版本

**如何提问**:

```
⚠️ 高风险操作警告

操作: [具体动作]
风险: [可能后果]
不可逆: [无法撤销的原因]

是否确认执行？

建议: [更安全的替代方案，如果有]
```

---

## ✅ 不应该问的问题（自行决策）

### 文件命名

**不要问**:

- "文件应该叫什么名字？"
- "放在哪个目录？"

**自行决策**:

- 遵循项目现有命名规范
- 如无规范，使用常见约定
- 例如: `user-service.ts`, `UserController.java`

---

### 代码风格

**不要问**:

- "用单引号还是双引号？"
- "缩进用 2 空格还是 4 空格？"
- "要不要加分号？"

**自行决策**:

- 检查项目中现有代码风格
- 遵循语言社区标准
- 例如: Python → PEP 8, JavaScript → Prettier

---

### 依赖版本

**不要问**:

- "应该用哪个版本的库？"
- "要不要升级这个依赖？"

**自行决策**:

- 使用项目现有依赖的版本
- 新依赖选择最新稳定版
- 如有兼容性问题，选择兼容版本

---

### 测试策略

**不要问**:

- "需要写单元测试吗？"
- "测试覆盖率要多少？"

**自行决策**:

- 遵循 `rules/domain/testing.md`
- 默认要求: 80% 覆盖率
- 关键路径必须测试

---

### UI 细节

**不要问**:

- "按钮用什么颜色？"
- "字体多大？"
- "间距多少？"

**自行决策**:

- 遵循设计规范（如果有）
- 无规范时使用常见标准
- 例如: 主按钮蓝色, 字体 16px, 间距 8px 倍数

---

## 🤔 判断标准

### 提问前问自己：

1. **能从代码库推断吗？**
   → 能 → 不要问

2. **有明确的最佳实践吗？**
   → 有 → 不要问

3. **是个人偏好问题吗？**
   → 是 → 不要问（自行选择）

4. **符合 4 种致命阻塞吗？**
   → 否 → 不要问

---

## 📝 决策优先级

遇到需要决策的情况：

```
1. 项目现有规范
   ↓ (如果没有)
2. 相关最佳实践
   ↓ (如果没有)
3. 语言/框架约定
   ↓ (如果没有)
4. 常见行业标准
   ↓ (如果没有)
5. 合理的默认选择
```

## 💡 示例对比

### ❌ 不好的提问

**场景**: 用户要求 "写一个用户登录 API"

**不要这样**:

```
问题1: 应该用什么数据库？
问题2: 密码加密用什么算法？
问题3: Token 过期时间设置多久？
问题4: 错误提示用中文还是英文？
问题5: API 路径是 /login 还是 /auth？
```

### ✅ 正确的做法

**直接实现**:

```
数据库: 用项目现有的（检查配置文件）
加密: bcrypt（行业标准）
Token 过期: 24小时（常见默认值）
错误提示: 跟项目其他 API 保持一致
API 路径: /auth/login（RESTful 约定）
```

**展示计划让用户确认** → 执行

---

### ✅ 合理的提问

**场景**: 用户要求 "清理所有测试数据"

**这样提问**:

```
⚠️ 高风险操作警告

操作: 删除数据库中所有 test_* 开头的记录
风险: 如果有生产数据也以 test_ 开头，会被误删
不可逆: 删除后无法恢复

是否确认执行？

建议: 先备份数据库，或者先 SELECT 确认要删除的记录
```

---

## 🎯 总结

**工作原则**:

```
自行决策 > 展示计划确认 > 提问

提问是最后的手段，只用于 4 种致命阻塞。
```

**好的工作模式**:

- 快速规划 → 展示 → 确认 → 执行
- 95% 的问题都能自行决策
- 5% 的真正阻塞才提问

**不好的工作模式**:

- 遇到任何小事都问
- 让用户做所有决策
- 浪费时间在不重要的细节上
