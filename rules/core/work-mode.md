# 核心工作模式

> **原则**: 计划 → 确认 → 执行到底 → 验收

---

## 🎯 四步工作流

```
1️⃣ 收到任务
   ↓ 创建 TodoList 规划
2️⃣ 展示计划
   ↓ 等待用户确认
3️⃣ 执行到底（不问问题）
   ↓ 完成所有任务
4️⃣ 总结验收
```

---

## ✅ Step 1: 收到任务 → 规划

### 自动创建 TodoList

使用 `TaskCreate` 工具记录任务：

```markdown
- [ ] 理解需求
- [ ] 设计方案
- [ ] 实现功能
- [ ] 测试验证
- [ ] 总结文档
```

### 何时需要规划？

- **需要**: 3+ 步骤的非平凡任务
- **不需要**: 单一、直接的简单任务

---

## 📋 Step 2: 展示计划 → 等待确认

### 展示内容

1. **任务分解**: 具体步骤列表
2. **预估难度**: 简单/中等/复杂
3. **潜在风险**: 如果有的话
4. **所需工具**: 使用什么技术/工具

### 等待明确确认

**继续的信号**:

- "好的"
- "开始吧"
- "继续"
- "ready"

**暂停的信号**:

- "等等"
- "先不要"
- "让我想想"

---

## 🚀 Step 3: 执行到底（不问问题）

### 核心原则

**不问问题，自行决策**

除非遇到 **4种致命阻塞**（见 `blocking-rules.md`），否则自行决策并执行。

### 自行决策的事项

✅ 文件命名
✅ 代码风格
✅ 依赖版本
✅ 测试策略
✅ UI 细节
✅ 函数命名
✅ 目录结构
✅ 注释风格

**遵循原则**: 现有规范 > 最佳实践 > 常见约定

### 执行期间

- **更新 TodoList**: 完成一项 → 标记完成
- **记录问题**: 遇到问题 → 记录到 notes.md
- **保持专注**: 不偏离原计划

---

## 📝 Step 4: 总结验收

### 总结内容

1. **完成情况**: 哪些做了，哪些没做
2. **遇到的问题**: 关键决策点
3. **重要发现**: 值得记录的经验
4. **后续建议**: 如果有的话

### 验收标准

- [ ] 所有 TodoList 项目完成
- [ ] 代码可以运行
- [ ] 测试通过
- [ ] 文档更新（如需要）

---

## 🔄 长任务的阶段门控

对于大型任务（>1小时），可以分阶段执行：

```
Phase 1: 需求理解
  ↓ [用户确认 "ready"]
Phase 2: 设计方案
  ↓ [用户确认 "ready"]
Phase 3: 实现代码
  ↓ [用户确认 "ready"]
Phase 4: 测试验证
```

### 阶段门控原则

**永远不进入下一阶段，直到用户明确确认。**

---

## 📁 三文件模式（长任务必用）

长任务（>30分钟）使用三文件模式：

```
task_plan.md     - 任务规划和进度追踪
notes.md         - 研究笔记和发现记录
[deliverable].md - 最终产出物
```

### 关键机制

**每个重要决策点前，重新读取 task_plan.md**

防止目标漂移，刷新注意力窗口。

---

## ⚠️ 失败追踪

记录所有失败，避免重复错误：

```markdown
## Errors Encountered

### [时间] 错误类型

**Error**: 具体错误信息
**Root Cause**: 根本原因
**Solution**: 解决方案
**Learning**: 经验教训
```

---

## 🎯 成功的工作模式

```
收到任务
  ↓
快速规划（TodoList）
  ↓
展示 + 等待确认
  ↓
执行到底（自行决策）
  ↓
总结验收（交付成果）
```

**不是**:

```
收到任务
  ↓
问一堆问题
  ↓
再问更多问题
  ↓
还在问问题...
```
